초기에는 Aider가 가장 널리 사용되는 도구였다. 모두 터미널 기반의 CLI 도구로 통합되는 과정에서

IDE에서 code completion을 이야기하는 사람은 없다. 변화의 과정에 있다."
#+end_quote

이 통찰은 AI 코딩 도구의 근본적 패러다임 전환을 짚고 있습니다.

* 패러다임의 진화

** 1세대: Code Completion (2021-2023)
*** 대표: GitHub Copilot

**** 특징
- 줄 단위 자동완성
- 함수/클래스 제안
- IDE 통합
- "똑똑한 자동완성"

**** 한계
- 수동적 지원
- 컨텍스트 제한
- 파일 단위 작업
- 개발자가 여전히 주도

**** 현재 상태
#+begin_quote
"IDE에서 code completion을 이야기하는 사람은 없다"
#+end_quote

- 더 이상 차별화 요소 아님
- 기본 기능으로 흡수됨
- 화두에서 사라짐

** 2세대: Pair Programming (2022-2024)
*** 대표: Aider

**** 특징
- 대화형 코딩
- Git 기반 워크플로우
- 파일 편집 자동화
- SWE-Bench 고득점

***** 아키텍처

두 가지 모델 협업:

결과: 85% 정확도 (최고 기록)
#+end_example

- **Architect**: 추론에 강한 모델 (설계)
- **Editor**: 코드 조작에 능한 모델 (구현)

***** 철학: "Pair Programming"
#+begin_quote
"AI pair programming in your terminal"
#+end_quote

- 개발자와 AI의 협업

- 명확한 버전 관리
- 추적 가능한 변경사항

**** 한계
- 여전히 개발자 중심
- 단일 에이전트 (Architect/Editor도 하나의 세션)
- Git 워크플로우에 제약
- 프로토콜 지원 부족

**** ACP 지원 상태

**중요**: Aider는 **ACP 네이티브 지원이 없습니다**.

**Agent Communication Protocol (ACP) 관련**:
- IBM BeeAI 환경에서 Aider 언급됨
- "다양한 오픈소스 에이전트 협업" 예시로 등장

**Agent Client Protocol (ACP) 관련**:
- **공식 지원**: 개발 중
- **커뮤니티 브릿지**: `jorgejhms/aider-acp`
  - Standalone ACP 에이전트로 실행
  - Zed와 JSON-RPC over stdio 통신
  - Aider CLI를 subprocess로 호출

**agent-shell에 Aider가 없는 이유**:
#+begin_quote
agent-shell은 ACP/A2A 네이티브 지원 에이전트만 포함합니다.
Aider는 커뮤니티 브릿지만 있고, 네이티브 지원이 없어 제외되었습니다.
#+end_quote

** 3세대: Vibe Coding (2025-)
*** 대표: Goose, OpenCode

**** 바이브 코딩이란?

**정의** (Andrej Karpathy, 2025년 2월 2일):
#+begin_quote
"자연어를 작동하는 코드로 변환, AI 에이전트의 도움으로.
개발자가 AI 에이전트에게 무언가를 만들도록 요청하고, **흐름을 따라간다**."
#+end_quote

**핵심 개념**:
- 개발자가 명시적으로 모든 것을 지시하지 않음
- AI 에이전트가 자율적으로 작업
- 직관적이고 반복적인 프로세스
- "Vibe" = 느낌, 분위기, 흐름

**** Goose의 바이브 코딩 구현

***** 1. 음성 기반 코딩
Max Novich (Block AI 팀) 데모:
- **키보드/마우스 없이** 전체 웹 앱 생성
- 음성 명령만으로
- "Speech MCP" 활용

***** 2. Subagent Orchestration
#+begin_example
개발자
  ↓ (요청)
Goose (Main Agent)
  ↓ (조율)
├─ Planner (기획)
├─ PM (관리)
├─ Architect (설계)
├─ Frontend Dev
├─ Backend Dev
├─ QA Engineer
└─ Tech Writer
#+end_example

**실제 사례**: AI BriefMe 앱 (1시간 완성)

**7개 에이전트 역할**:
1. **Planner**: 제품 비전, MVP 범위 정의
2. **PM**: 작업 분해 및 조율
3. **Architect**: 프로젝트 구조 및 기술 스택
4. **Frontend Dev**: UI/UX 구현
5. **Backend Dev**: API 로직 구축
6. **QA Engineer**: 테스트 및 문제 파악
7. **Tech Writer**: 문서화

**결과**:
- ✅ 1시간 내 완전 작동하는 웹 앱
- ⚠️ QA 평가: "프로덕션 준비 상태 아님"
- 🔧 보안, 성능 개선 필요

***** 3. Headless Goose
- 백엔드가 프로그래매틱하게 Goose 호출
- API 기반 자동화
- CI/CD 통합 가능

**** 책임 있는 바이브 코딩

Goose가 제시한 9가지 안전 기능:

**1. `.gooseignore`**
#+begin_src
# .gitignore처럼 작동
# AI가 읽거나 수정하지 않을 파일
.env
secrets/
credentials.json
#+end_src

**2. `/plan` 명령**
- 코드 변경 **전에** 구조화된 분석 제공
- 의도 확인
- 영향 범위 파악

**3. 모드 선택**
| 모드 | 설명 | 사용 시나리오 |
|------|------|---------------|
| Chat | 질문만 | 탐색, 학습 |
| Approval | 모든 작업 승인 | 중요 프로젝트 |
| Smart Approval | 안전한 작업 자동 | 일반 개발 |
| Autonomous | 완전 자율 | 실험, 프로토타입 |

**4. 버전 관리**
- Git 커밋 자동화
- 롤백 가능
- 변경 추적

**5. 비판적 사고**
- AI 제안을 무조건 수용하지 않음
- 검토 및 이해
- 학습 기회

**6. `.goosehints` 파일**
- 프로젝트별 지침
- 코딩 스타일
- 제약사항

**7. CI/CD 통합**
- 자동 테스트
- 린트 검사
- 품질 게이트

**8. MCP 서버 허용 목록**
- 신뢰할 수 있는 MCP 서버만
- 보안 위험 최소화

**9. 고성능 LLM 선택**
- Claude Sonnet 4 등
- 품질 우선

**철학**:
#+begin_quote
"창의적 자유와 프로젝트 안전성의 균형"
#+end_quote

**** OpenCode의 바이브 코딩

**Multi-session workflows**:
- 여러 에이전트가 동일 프로젝트에서 병렬 작업
- 충돌 없이 협업
- 진정한 다중 에이전트 시스템

**75+ LLM 지원**:
- 작업별 최적 모델 선택
- 유연성 극대화
- 비용 최적화

**터미널 중심 설계**:
- Arch Linux 사용자 같은 "진정한 터미널 유저"를 위함
- 고품질 도구
- 개발자 경험 우선

**** 패러다임 차이: Pair Programming vs Vibe Coding

| 항목 | Pair Programming (Aider) | Vibe Coding (Goose/OpenCode) |
|------|--------------------------|------------------------------|
| **주도권** | 개발자 | AI + 개발자 협업 |
| **에이전트** | 단일 (Architect/Editor) | 다중 (오케스트레이션) |
| **워크플로우** | Git 중심 | 흐름 중심 |
| **명시성** | 명확한 지시 | "흐름 따라가기" |
| **프로토콜** | 없음 (커뮤니티 브릿지) | ACP/MCP 네이티브 |
| **IDE 통합** | 코멘트 기반 | 독립적 (MCP 통해 가능) |
| **철학** | 협업 파트너 | 자율 에이전트 팀 |

** 4세대: Agent Orchestration (미래)

*** 미래 방향

**** Autonomous Dev Teams
#+begin_example
Product Manager Agent
  ↓
Architect Agent → Frontend Agent
  ↓              ↓
Backend Agent  → QA Agent
  ↓              ↓
DevOps Agent   → Security Agent
#+end_example

**** Self-improving Systems

- 메타 학습
- 진화하는 워크플로우

**** Human-in-the-loop
- 전략적 결정만 인간
- 전술적 실행은 에이전트

* 결론: 변화의 과정

** 현재 상황

우리는 **3세대 (Vibe Coding)**로 전환하는 중입니다:

*** 1세대: Code Completion
- 기본 기능으로 흡수됨
- 더 이상 화두 아님
- ✅ 완료

*** 2세대: Pair Programming
- Aider로 대표됨
- 여전히 강력하고 유용
- 하지만 "과도기" 기술
- ⏳ 과도기

*** 3세대: Vibe Coding
- Goose/OpenCode로 구현
- Multi-agent orchestration
- 프로토콜 표준화 (ACP/A2A)
- 🔥 현재 진행형

*** 4세대: Agent Orchestration
- 자율적 개발 팀
- 역할별 전문화
- 인간은 감독만
- 🚀 미래

** agent-shell의 의미

agent-shell은 **3세대 → 4세대로 가는 다리**입니다:

- Emacs: 인간 인터페이스
- ACP/A2A: 에이전트 간 통신
- OpenCode/Goose/Qwen: 다양한 에이전트
- 통합 관리: 단일 허브

Aider를 포함하지 않은 것은 **의도적 선택**:
- 2세대 패러다임 (Pair Programming)
- ACP 네이티브 부재
- 3세대 방향과 불일치

** "개발 도구"에서 "바이브 코딩 도구"로

#+begin_quote
"모두 터미널 기반의 CLI 도구로 통합되는 과정에서
**개발 도구**라기보다는 **바이브 코딩 도구**로 진화하고 있다."
#+end_quote

이 통찰이 정확한 이유:

*** 개발 도구 (전통)
- 명시적 제어
- 도구는 수동적
- 개발자가 모든 결정
- 예측 가능성

*** 바이브 코딩 도구 (새로운)
- "흐름 따라가기"
- AI 자율성
- 협업적 결정
- 창의적 발견

** 앞으로의 방향

*** 단기 (2025)
- Vibe Coding 성숙화
- 프로토콜 표준화 완료
- agent-shell 같은 통합 도구 증가

*** 중기 (2026-2027)
- Multi-agent systems 일반화
- 역할별 전문 에이전트
- 프로덕션 준비 완료

*** 장기 (2028+)
- 자율 개발 팀
- 인간은 전략만
- AI 에이전트가 대부분 실행

** 핵심 메시지

#+begin_quote
"변화의 과정에 있다"
#+end_quote

지금 우리는:
1. Code Completion을 당연하게 여기고
2. Pair Programming을 여전히 사용하지만
3. Vibe Coding으로 이동하고 있으며
4. Agent Orchestration을 준비하고 있습니다

Aider는 훌륭한 2세대 도구입니다.
OpenCode/Goose는 3세대를 열고 있습니다.
agent-shell은 3세대와 4세대를 연결합니다.

**선택은 당신이 어느 세대에 서고 싶은지에 달렸습니다.**
