* LSP vs MCP: 완전히 다른 레이어

** 핵심 차이

| 측면 | LSP | MCP |
|------|-----|-----|
| **정식 명칭** | Language Server Protocol | Model Context Protocol |
| **개발자** | Microsoft (2016) | Anthropic (2024) |
| **연결 대상** | IDE ↔ Language Server | LLM ↔ Tools/Data |
| **목적** | 코드 인텔리전스 | AI 워크플로우 |
| **정보 소스** | 컴파일러/파서 | 외부 도구/API |
| **토큰 사용** | **없음** (로컬 분석) | 있음 (LLM 호출) |
| **실행 모델** | 반응형 (Reactive) | 자율형 (Autonomous) |

** 해결하는 문제

*** LSP (2016년 Microsoft)
**문제**:
- N개의 IDE × M개의 언어 = N×M 통합 필요
- 각 IDE마다 언어별 플러그인 개발

**해결**:
- IDE와 Language Server 사이의 표준 프로토콜
- 한 번의 Language Server 구현으로 모든 IDE 지원

**영향**:
#+begin_quote
"LSP 덕분에 프로그래밍 언어 지원이 어느 때보다 좋아졌다"
#+end_quote

*** MCP (2024년 Anthropic)
**문제**:
- N개의 LLM 클라이언트 × M개의 도구/API = N×M 통합 필요
- 각 AI 앱마다 도구별 통합 개발

**해결**:
- LLM과 외부 리소스 사이의 표준 프로토콜
- 한 번의 MCP 서버 구현으로 모든 AI 클라이언트 지원

**영향**:
#+begin_quote
"MCP는 AI 시스템이 도구 및 데이터 소스와 통합하는 방식을 표준화"
#+end_quote

**** LSP: Reactive (반응형)
#+begin_src

  ↓

- 동기적 응답

**** MCP: Autonomous (자율형)

  ↓

  ↓

  ↓

#+end_src

- AI의 자율적 워크플로우
- 장기 실행 작업
- 비동기적 상호작용

** 핵심 가치: 토큰 제로

"LSP는 토큰 소모 없이 인간의 코딩 경험을 이어주는 것"

이 표현이 **정확한** 이유:

*** 전통적 방법 (RAG 등)
#+begin_example
질문: "이 함수의 타입은?"

1. 코드베이스 검색 (벡터 DB)
2. 관련 코드 청크 수집
3. LLM에 전송 (토큰 소모)
   ↓ Input: 1000+ 토큰
4. LLM 분석 및 추론
   ↓ Output: 100+ 토큰
5. 답변

비용: 1100+ 토큰
정확도: 추측 기반
#+end_example

*** LSP 방법
#+begin_example
질문: "이 함수의 타입은?"

1. LSP 서버에 hover 요청
   ↓ 로컬 통신
2. 컴파일러가 정확한 타입 반환
   ↓ 0 토큰
3. 답변

비용: 0 토큰
정확도: 컴파일러 수준 (100%)
#+end_example

** LSP가 제공하는 기능

*** 1. Hover Information
파일 위치의 심볼에 대한 정보:
- 타입 서명
- 문서 문자열
- 파라미터 정보

예시:
#+begin_src python
def process_data(data: List[Dict[str, Any]]) -> pd.DataFrame:
    """Process raw data into DataFrame."""
    ...

# Hover on 'process_data':
# → 타입: (data: List[Dict[str, Any]]) -> pd.DataFrame
# → 문서: "Process raw data into DataFrame."
#+end_src

**토큰 소모**: 0

*** 2. Go to Definition
심볼의 정의 위치 찾기:
- 함수 정의
- 클래스 정의
- 변수 선언

예시:
#+begin_src
user.save()
     ^^^^
# "save" 정의로 이동
# → models.py:45
#+end_src

**토큰 소모**: 0

*** 3. Find References
코드베이스 전체에서 사용 위치:
- 함수 호출
- 변수 참조
- 타입 사용

예시:
#+begin_src
# "User" 클래스의 모든 사용 찾기
→ 23개 파일, 187개 참조 발견
#+end_src

**토큰 소모**: 0

*** 4. Code Completion
문맥 인식 자동완성:
- 메서드 제안
- 변수 제안
- Import 제안

예시:
#+begin_src python
user.
     ↓
# 제안:
# - .id
# - .name
# - .email
# - .save()
# - .delete()
#+end_src

**토큰 소모**: 0

*** 5. Diagnostics
컴파일러 오류 및 경고:
- 타입 오류
- 문법 오류
- 린팅 경고

예시:
#+begin_src python
def add(a: int, b: int) -> int:
    return str(a + b)  # ← 오류!
    # Expected 'int', got 'str'
#+end_src

**토큰 소모**: 0

*** 6. Code Actions
리팩토링 및 빠른 수정:
- Extract function
- Rename symbol
- Import 추가
- Unused import 제거

**토큰 소모**: 0

** 왜 "인간의 코딩 경험"인가?

LSP가 제공하는 것은 **IDE가 인간에게 제공하는 것과 동일**:
- VS Code의 IntelliSense
- JetBrains의 code insight
- Emacs의 lsp-mode

**AI 에이전트도 같은 경험**을 얻습니다:
- 타입 정보
- 문서
- 정의 위치
- 참조 목록
- 오류 진단

**차이**: 인간은 시각적으로, AI는 프로그래매틱하게

** 컴파일러 수준의 정확도

*** RAG/LLM 방식
- 코드를 텍스트로 취급
- 패턴 매칭 및 추론
- 확률적 답변
- **오류 가능성**

*** LSP 방식
- 컴파일러/파서가 분석
- 정확한 AST(Abstract Syntax Tree)
- 타입 체크 결과
- **100% 정확**

예시:
#+begin_src python
# 복잡한 제네릭 타입
def transform<T extends BaseModel>(
    data: Array<T>,
    mapper: (item: T) => Partial<T>
): Array<Partial<T>> { ... }
#+end_src

- RAG/LLM: 추측 필요, 틀릴 수 있음
- LSP: 컴파일러가 정확히 알고 있음

* lsp-mcp 브릿지: 두 세계의 연결

** 문제 상황

- **LSP**: IDE용으로 설계됨
- **MCP**: AI 에이전트용
- AI 에이전트가 LSP 정보에 접근하려면?

**해결책**: lsp-mcp 서버

** lsp-mcp란?

#+begin_quote
"LSP 서버를 MCP 프로토콜로 프록시하여,
MCP 클라이언트(AI)가 IDE급 언어 인텔리전스를 쿼리할 수 있게 함"
#+end_quote

** 아키텍처

#+begin_example
AI 에이전트 (Claude, GPT 등)
  ↓ MCP 프로토콜
lsp-mcp 서버 (브릿지)
  ↓ LSP 프로토콜
Language Server (rust-analyzer, pyright 등)
  ↓ 파싱/분석
코드베이스
#+end_example

** 작동 방식

1. **AI 에이전트**: MCP 요청 전송
   #+begin_src json
   {
     "method": "get_info_on_location",
     "params": {
       "file": "src/main.rs",
       "line": 42,
       "character": 15
     }
   }
   #+end_src

2. **lsp-mcp**: LSP 요청으로 변환
   #+begin_src json
   {
     "method": "textDocument/hover",
     "params": {
       "textDocument": { "uri": "file:///src/main.rs" },
       "position": { "line": 42, "character": 15 }
     }
   }
   #+end_src

3. **Language Server**: 정확한 정보 반환
   #+begin_src json
   {
     "contents": {
       "kind": "markdown",
       "value": "```rust\nfn process_data(data: &[u8]) -> Result<String>\n```\nProcesses binary data..."
     }
   }
   #+end_src

4. **lsp-mcp**: MCP 응답으로 변환
5. **AI 에이전트**: 정보 수신 (0 토큰 소모)

** 주요 구현체

| 구현체 | 언어 | 강점 |
|--------|------|------|
| **Tritlo/lsp-mcp** | TypeScript | 우수한 문서화, 기능 풍부 |
| **nzrsky/lsp-mcp-server** | Zig | 높은 성능, 낮은 리소스 |

Claude Desktop 설정 (~/.config/claude/claude_desktop_config.json):

전통적 방법 (RAG):
#+begin_example
100개 파일에서 'User' 클래스 참조 찾기

1. 벡터 DB 검색
2. 관련 파일 100개 수집
3. LLM에 전송: 50,000 토큰
4. LLM 분석: 5,000 토큰 출력

총 비용: 55,000 토큰
예상 비용: $0.55 (GPT-4o)
#+end_example

LSP 방법:
#+begin_example
1. get_references("User")
2. 즉시 정확한 187개 참조 반환

총 비용: 0 토큰
예상 비용: $0.00
#+end_example

**절감**: 100%

**** 누적 효과
하루 100번 코드 분석:
- RAG: $55/일 = $1,650/월
- LSP: $0/일 = $0/월

**절감**: $1,650/월

** 워크플로우 최적화

*** Before (LSP 없이)
#+begin_example
1. 사용자: "User 클래스를 수정해줘"
   ↓
2. AI: 전체 코드베이스 벡터 검색 (토큰 소모)
   ↓
3. AI: 관련 파일 추측 (부정확할 수 있음)
   ↓
4. AI: LLM으로 영향 범위 분석 (토큰 대량 소모)
   ↓
5. AI: 수정 시도 (오류 가능)
   ↓
6. 사용자: 오류 발견, 재시도

비용: 높음
정확도: 중간
시간: 느림
#+end_example

*** After (LSP 활용)
#+begin_example
1. 사용자: "User 클래스를 수정해줘"
   ↓
2. AI: get_definition("User") (토큰 0)
   → models/user.py:15
   ↓
3. AI: get_references("User") (토큰 0)
   → 23개 파일, 187개 참조
   ↓
4. AI: 각 참조의 타입 정보 확인 (토큰 0)
   → 정확한 영향 범위 파악
   ↓
5. AI: 수정 (정확함)
   ↓
6. AI: get_diagnostics() (토큰 0)
   → 오류 없음 확인

비용: 최소 (LLM은 수정 생성만)
정확도: 높음 (컴파일러 수준)
시간: 빠름 (로컬 LSP)
#+end_example

**** 제안된 VSCode LSP API 목록
Issue #5495에서 요청한 구체적인 기능들:

**1. Symbol Navigation (심볼 탐색)**
- ~Go to definition~: 정의로 즉시 이동
- ~Find all references~: 모든 참조 위치 목록
- ~Workspace symbol search~: 전체 워크스페이스 심볼 검색

현재 Claude Code 방식:
#+begin_example
grep -r "functionName" . | # 텍스트 검색
  수동으로 파일 읽기 | # 여러 파일 오픈
  LLM이 추론 # 토큰 소모
#+end_example

LSP 방식:
#+begin_example
textDocument/definition → 즉시 정확한 위치 반환 (0 토큰)
#+end_example

**2. Type Intelligence (타입 인텔리전스)**
- ~Navigate type definitions~: 타입 정의 탐색
- ~Find implementations~: 인터페이스 구현체 찾기
- ~Type hierarchy~: 타입 계층 구조

TypeScript 예시:
#+begin_src typescript
interface User {
  id: number;
  name: string;
}

// LSP로 즉시 찾기:
// 1. User 인터페이스 정의
// 2. User를 구현한 모든 클래스
// 3. User를 사용하는 모든 함수
#+end_src

**3. Search Capabilities (스마트 검색)**
- ~Smart file filtering~: .gitignore 존중
- ~Symbol-aware search~: 심볼 기반 검색
- ~Semantic search~: 의미론적 검색

현재 문제:
- glob으로 모든 파일 스캔
- .gitignore 무시 가능
- node_modules까지 검색

LSP 방식:
- 인덱스된 심볼만 검색
- 프로젝트 설정 존중
- 즉시 결과 반환

**4. Code Intelligence (코드 인텔리전스)**
- ~Hover information~: 타입/문서 정보
- ~Call hierarchies~: 호출 관계 트리
- ~Rename operations~: 안전한 리팩토링

Hover 예시:
#+begin_src python
def process_user(user: User) -> None:
    # ← hover on 'user'

    # - Type: User
    # - Properties: id, name, email
    # - Methods: save(), delete()
    # 토큰 소모: 0
#+end_src

**5. Diagnostics (진단)**
- ~Real-time errors~: 실시간 오류 감지
- ~Type checking~: 타입 검사
- ~Linting warnings~: 린팅 경고

Claude Code 현재:
- 코드 작성 후 실행해야 오류 발견

**설정 예시** (~/.emacs.d/init.el):
#+begin_src elisp
;; 1. lsp-mode: Emacs 네이티브 LSP
(use-package lsp-mode
  :ensure t
  :hook
  ((rust-mode . lsp)
   (python-mode . lsp)
   (typescript-mode . lsp))
  :config
  (setq lsp-rust-analyzer-server-command '("rust-analyzer"))
  (setq lsp-pyright-server-command '("pyright-langserver" "--stdio")))

;; 2. agent-shell: 다중 AI 에이전트
(use-package agent-shell
  :ensure t
  :bind
  ("C-c a o" . agent-shell-opencode)    ; OpenCode (LSP 네이티브)
  ("C-c a c" . agent-shell-claude-code) ; Claude Code
  :config
  ;; OpenCode: 빠른 코드 작업
  (setq agent-shell-opencode-use-cases
        '("리팩토링" "참조 찾기" "타입 확인" "코드 탐색"))

  ;; Claude Code: 복잡한 작업
  (setq agent-shell-claude-code-use-cases
        '("아키텍처 설계" "복잡한 버그 수정" "알고리즘 최적화")))

;; 3. 스마트 에이전트 선택
(defun my/smart-agent-dispatch ()
  "작업 유형에 따라 적절한 에이전트 선택"
  (interactive)
  (let ((task (read-string "작업: ")))
    (cond
     ;; LSP 기반 작업 → OpenCode
     ((string-match-p "찾기\\|리팩토링\\|이름변경" task)
      (agent-shell-opencode task))

     ;; 복잡한 추론 → Claude Code
     ((string-match-p "설계\\|최적화\\|버그" task)
      (agent-shell-claude-code task))

     ;; 기본: OpenCode (비용 효율)
     (t (agent-shell-opencode task)))))

(global-set-key (kbd "C-c a a") 'my/smart-agent-dispatch)
#+end_src

1. 일상적인 코딩: Emacs lsp-mode (무료, 즉시)

#+end_example
