* LSP vs MCP: 완전히 다른 레이어

** 핵심 차이

| 측면 | LSP | MCP |
|------|-----|-----|
| **정식 명칭** | Language Server Protocol | Model Context Protocol |
| **개발자** | Microsoft (2016) | Anthropic (2024) |
| **연결 대상** | IDE ↔ Language Server | LLM ↔ Tools/Data |
| **목적** | 코드 인텔리전스 | AI 워크플로우 |
| **정보 소스** | 컴파일러/파서 | 외부 도구/API |
| **토큰 사용** | **없음** (로컬 분석) | 있음 (LLM 호출) |
| **실행 모델** | 반응형 (Reactive) | 자율형 (Autonomous) |

** 해결하는 문제

*** LSP (2016년 Microsoft)
**문제**:
- N개의 IDE × M개의 언어 = N×M 통합 필요
- 각 IDE마다 언어별 플러그인 개발

**해결**:
- IDE와 Language Server 사이의 표준 프로토콜
- 한 번의 Language Server 구현으로 모든 IDE 지원

**영향**:
#+begin_quote
"LSP 덕분에 프로그래밍 언어 지원이 어느 때보다 좋아졌다"
#+end_quote

*** MCP (2024년 Anthropic)
**문제**:
- N개의 LLM 클라이언트 × M개의 도구/API = N×M 통합 필요
- 각 AI 앱마다 도구별 통합 개발

**해결**:
- LLM과 외부 리소스 사이의 표준 프로토콜
- 한 번의 MCP 서버 구현으로 모든 AI 클라이언트 지원

**영향**:
#+begin_quote
"MCP는 AI 시스템이 도구 및 데이터 소스와 통합하는 방식을 표준화"
#+end_quote

** 기술적 유사점과 차이점

*** 유사점
- 둘 다 JSON-RPC 2.0 기반
- 메시지 플로우 패턴 재사용
- Many-to-many 문제 해결

*** 결정적 차이

**** LSP: Reactive (반응형)
#+begin_src
사용자 입력 (타이핑)
  ↓
IDE 요청
  ↓
Language Server 응답 (즉시)
  ↓
자동완성/진단 표시
#+end_src

- IDE의 요청에 반응
- 사용자 입력 기반
- 동기적 응답

**** MCP: Autonomous (자율형)
#+begin_src
AI 에이전트 계획
  ↓
MCP 도구 호출 (여러 번)
  ↓
결과 수집 및 분석
  ↓
다음 액션 결정 (자율적)
#+end_src

- AI의 자율적 워크플로우
- 장기 실행 작업
- 비동기적 상호작용

* LSP가 제공하는 것: "토큰 소모 없는 코드 인텔리전스"

** 핵심 가치: 토큰 제로

#+begin_quote
"LSP는 토큰 소모 없이 인간의 코딩 경험을 이어주는 것"
#+end_quote

이 표현이 **정확한** 이유:

*** 전통적 방법 (RAG 등)
#+begin_example
질문: "이 함수의 타입은?"

1. 코드베이스 검색 (벡터 DB)
2. 관련 코드 청크 수집
3. LLM에 전송 (토큰 소모)
   ↓ Input: 1000+ 토큰
4. LLM 분석 및 추론
   ↓ Output: 100+ 토큰
5. 답변

비용: 1100+ 토큰
정확도: 추측 기반
#+end_example

*** LSP 방법
#+begin_example
질문: "이 함수의 타입은?"

1. LSP 서버에 hover 요청
   ↓ 로컬 통신
2. 컴파일러가 정확한 타입 반환
   ↓ 0 토큰
3. 답변

비용: 0 토큰
정확도: 컴파일러 수준 (100%)
#+end_example

** LSP가 제공하는 기능

*** 1. Hover Information
파일 위치의 심볼에 대한 정보:
- 타입 서명
- 문서 문자열
- 파라미터 정보

예시:
#+begin_src python
def process_data(data: List[Dict[str, Any]]) -> pd.DataFrame:
    """Process raw data into DataFrame."""
    ...

# Hover on 'process_data':
# → 타입: (data: List[Dict[str, Any]]) -> pd.DataFrame
# → 문서: "Process raw data into DataFrame."
#+end_src

**토큰 소모**: 0

*** 2. Go to Definition
심볼의 정의 위치 찾기:
- 함수 정의
- 클래스 정의
- 변수 선언

예시:
#+begin_src
user.save()
     ^^^^
# "save" 정의로 이동
# → models.py:45
#+end_src

**토큰 소모**: 0

*** 3. Find References
코드베이스 전체에서 사용 위치:
- 함수 호출
- 변수 참조
- 타입 사용

예시:
#+begin_src
# "User" 클래스의 모든 사용 찾기
→ 23개 파일, 187개 참조 발견
#+end_src

**토큰 소모**: 0

*** 4. Code Completion
문맥 인식 자동완성:
- 메서드 제안
- 변수 제안
- Import 제안

예시:
#+begin_src python
user.
     ↓
# 제안:
# - .id
# - .name
# - .email
# - .save()
# - .delete()
#+end_src

**토큰 소모**: 0

*** 5. Diagnostics
컴파일러 오류 및 경고:
- 타입 오류
- 문법 오류
- 린팅 경고

예시:
#+begin_src python
def add(a: int, b: int) -> int:
    return str(a + b)  # ← 오류!
    # Expected 'int', got 'str'
#+end_src

**토큰 소모**: 0

*** 6. Code Actions
리팩토링 및 빠른 수정:
- Extract function
- Rename symbol
- Import 추가
- Unused import 제거

**토큰 소모**: 0

** 왜 "인간의 코딩 경험"인가?

LSP가 제공하는 것은 **IDE가 인간에게 제공하는 것과 동일**:
- VS Code의 IntelliSense
- JetBrains의 code insight
- Emacs의 lsp-mode

**AI 에이전트도 같은 경험**을 얻습니다:
- 타입 정보
- 문서
- 정의 위치
- 참조 목록
- 오류 진단

**차이**: 인간은 시각적으로, AI는 프로그래매틱하게

** 컴파일러 수준의 정확도

*** RAG/LLM 방식
- 코드를 텍스트로 취급
- 패턴 매칭 및 추론
- 확률적 답변
- **오류 가능성**

*** LSP 방식
- 컴파일러/파서가 분석
- 정확한 AST(Abstract Syntax Tree)
- 타입 체크 결과
- **100% 정확**

예시:
#+begin_src python
# 복잡한 제네릭 타입
def transform<T extends BaseModel>(
    data: Array<T>,
    mapper: (item: T) => Partial<T>
): Array<Partial<T>> { ... }
#+end_src

- RAG/LLM: 추측 필요, 틀릴 수 있음
- LSP: 컴파일러가 정확히 알고 있음

* lsp-mcp 브릿지: 두 세계의 연결

** 문제 상황

- **LSP**: IDE용으로 설계됨
- **MCP**: AI 에이전트용
- AI 에이전트가 LSP 정보에 접근하려면?

**해결책**: lsp-mcp 서버

** lsp-mcp란?

#+begin_quote
"LSP 서버를 MCP 프로토콜로 프록시하여,
MCP 클라이언트(AI)가 IDE급 언어 인텔리전스를 쿼리할 수 있게 함"
#+end_quote

** 아키텍처

#+begin_example
AI 에이전트 (Claude, GPT 등)
  ↓ MCP 프로토콜
lsp-mcp 서버 (브릿지)
  ↓ LSP 프로토콜
Language Server (rust-analyzer, pyright 등)
  ↓ 파싱/분석
코드베이스
#+end_example

** 작동 방식

1. **AI 에이전트**: MCP 요청 전송
   #+begin_src json
   {
     "method": "get_info_on_location",
     "params": {
       "file": "src/main.rs",
       "line": 42,
       "character": 15
     }
   }
   #+end_src

2. **lsp-mcp**: LSP 요청으로 변환
   #+begin_src json
   {
     "method": "textDocument/hover",
     "params": {
       "textDocument": { "uri": "file:///src/main.rs" },
       "position": { "line": 42, "character": 15 }
     }
   }
   #+end_src

3. **Language Server**: 정확한 정보 반환
   #+begin_src json
   {
     "contents": {
       "kind": "markdown",
       "value": "```rust\nfn process_data(data: &[u8]) -> Result<String>\n```\nProcesses binary data..."
     }
   }
   #+end_src

4. **lsp-mcp**: MCP 응답으로 변환
5. **AI 에이전트**: 정보 수신 (0 토큰 소모)

** 주요 구현체

| 구현체 | 언어 | 강점 |
|--------|------|------|
| **Tritlo/lsp-mcp** | TypeScript | 우수한 문서화, 기능 풍부 |
| **nzrsky/lsp-mcp-server** | Zig | 높은 성능, 낮은 리소스 |

** 설정 예시

Claude Desktop 설정 (~/.config/claude/claude_desktop_config.json):

{
  "mcpServers": {
    "lsp-rust": {

* LSP의 전략적 중요성

** AI 에이전트에게 LSP가 중요한 이유

*** 1. 비용 절감 (토큰 제로)

**** 예시: 대규모 리팩토링
전통적 방법 (RAG):
#+begin_example
100개 파일에서 'User' 클래스 참조 찾기

1. 벡터 DB 검색
2. 관련 파일 100개 수집
3. LLM에 전송: 50,000 토큰
4. LLM 분석: 5,000 토큰 출력

총 비용: 55,000 토큰
예상 비용: $0.55 (GPT-4o)
#+end_example

LSP 방법:
#+begin_example
1. get_references("User")
2. 즉시 정확한 187개 참조 반환

총 비용: 0 토큰
예상 비용: $0.00
#+end_example

**절감**: 100%

**** 누적 효과
하루 100번 코드 분석:
- RAG: $55/일 = $1,650/월
- LSP: $0/일 = $0/월

**절감**: $1,650/월

*** 2. 정확도 향상

**** 타입 정보
#+begin_src typescript
// 복잡한 타입
type ApiResponse<T> = {
  data: T | null;
  error: Error | null;
  meta: {
    timestamp: number;
    version: string;
  }
}

function handleResponse<T>(res: ApiResponse<T>): T {
  // LLM: res의 정확한 타입 추론 어려움
  // LSP: 정확히 ApiResponse<T> 알고 있음
}
#+end_src

**LSP**: 100% 정확
**LLM**: 추측 기반

**** 리팩토링 안전성
함수명 변경 시:
- LLM: 문자열 매칭, 주석도 변경할 수 있음
- LSP: 실제 참조만 정확히 찾음

*** 3. 속도 향상

| 작업 | RAG/LLM | LSP |
|------|---------|-----|
| **정의 찾기** | 5-10초 (LLM 호출) | <100ms (로컬) |
| **참조 찾기** | 10-30초 (대규모 검색) | <500ms |
| **타입 정보** | 5초 (컨텍스트 전송) | <50ms |

**속도 향상**: 10-100배

*** 4. 대규모 코드베이스

**** 문제
- 100만 줄 코드베이스
- LLM 컨텍스트 한계 (256K 토큰도 부족)
- RAG로 정확한 정보 찾기 어려움

**** LSP 해결책
- Language Server가 전체 코드베이스 인덱싱
- 즉시 정확한 정보 제공
- 컨텍스트 한계 없음

*** 5. 오프라인 작업

- LSP: 로컬에서 작동
- LLM: API 호출 필요

**장점**:
- 네트워크 불필요

** 언제 LSP가 필수인가?

*** 필수 시나리오

**** 1. 타입 안전 언어
- TypeScript, Rust, Go, Java 등
- 타입 정보 없이는 정확한 리팩토링 불가능

**** 2. 대규모 리팩토링
- 함수/클래스명 변경
- API 시그니처 변경
- 모듈 재구성

**** 3. 비용 민감 프로젝트
- 토큰 비용 최소화
- 대량 코드 분석

**** 4. 실시간 피드백
- 코딩 중 즉시 오류 확인
- 자동완성
- 린팅

*** 선택적 시나리오

**** 1. 탐색적 분석
- 아키텍처 이해
- 코드 패턴 파악
- LLM의 추론이 유용

**** 2. 동적 언어 (Python, Ruby)
- 타입 정보 제한적
- LSP 이점 감소
- 하지만 여전히 유용 (정의, 참조)

**** 3. 문서/주석 작성
- LSP 불필요
- LLM이 더 적합

** OpenCode의 차별화: LSP + MCP

*** 강점

**** 1. 이중 프로토콜
#+begin_example
OpenCode
  ├─ LSP (코드 인텔리전스)
  │   → 토큰 제로
  │   → 컴파일러 정확도
  └─ MCP (도구 접근)
      → 외부 리소스
      → 확장 가능성
#+end_example

**** 2. 최적 워크플로우
#+begin_example
1. LSP로 코드 구조 파악 (토큰 0)
   ↓
2. 필요시 MCP 도구 호출
   ↓
3. 최소 토큰으로 LLM 쿼리
#+end_example

**** 3. 비용 효율
- 토큰 소모 최소화
- 정확도는 최대화
- Claude Code 대비 우위 가능

*** Claude Code와의 비교

| 항목 | OpenCode (LSP+MCP) | Claude Code (추정) |
|------|-------------------|-------------------|
| **코드 분석** | LSP (토큰 0) | LLM? (토큰 소모) |
| **타입 정보** | LSP (100% 정확) | 추론? (추측) |
| **비용** | 최소화 | 높음 (추정) |
| **속도** | 빠름 (로컬) | 느림 (API) |
| **정확도** | 컴파일러 수준 | LLM 추론 |

**주의**: Claude Code의 실제 구현은 확인 필요

*** 4. 사용

**** AI 에이전트에게 지시
#+begin_quote
"get_info_on_location을 사용해서 이 함수의 타입을 확인해줘."

"get_references로 User 클래스가 어디서 사용되는지 찾아줘."

"get_diagnostics로 이 파일의 오류를 확인해줘."
#+end_quote

**** 자동 활용
에이전트가 자동으로:
1. 코드 편집 전 LSP로 타입 확인
2. 참조 찾아 영향 범위 파악
3. 변경 후 진단으로 오류 체크

** 워크플로우 최적화

*** Before (LSP 없이)
#+begin_example
1. 사용자: "User 클래스를 수정해줘"
   ↓
2. AI: 전체 코드베이스 벡터 검색 (토큰 소모)
   ↓
3. AI: 관련 파일 추측 (부정확할 수 있음)
   ↓
4. AI: LLM으로 영향 범위 분석 (토큰 대량 소모)
   ↓
5. AI: 수정 시도 (오류 가능)
   ↓
6. 사용자: 오류 발견, 재시도

비용: 높음
정확도: 중간
시간: 느림
#+end_example

*** After (LSP 활용)
#+begin_example
1. 사용자: "User 클래스를 수정해줘"
   ↓
2. AI: get_definition("User") (토큰 0)
   → models/user.py:15
   ↓
3. AI: get_references("User") (토큰 0)
   → 23개 파일, 187개 참조
   ↓
4. AI: 각 참조의 타입 정보 확인 (토큰 0)
   → 정확한 영향 범위 파악
   ↓
5. AI: 수정 (정확함)
   ↓
6. AI: get_diagnostics() (토큰 0)
   → 오류 없음 확인

비용: 최소 (LLM은 수정 생성만)
정확도: 높음 (컴파일러 수준)
시간: 빠름 (로컬 LSP)
#+end_example

** 실전 예시

*** 예시 1: 타입 안전 리팩토링

**** 작업
#+begin_src typescript
// 기존
interface User {
  name: string;
  email: string;
}

// 변경: age 필드 추가
interface User {
  name: string;
  email: string;
  age: number; // 새 필드
}
#+end_src

**** LSP 활용 워크플로우
1. ~get_references("User")~
   → 모든 User 사용 위치 (47개)

2. 각 위치에서 ~get_info_on_location~
   → 어떻게 사용되는지 확인

3. 수정 필요한 곳 식별:
   #+begin_src typescript
   // 이런 곳들:
   const user: User = { name: "Alice", email: "..." };
   // → age 추가 필요
   #+end_src

4. 수정 후 ~get_diagnostics~
   → 타입 오류 확인

5. 모든 오류 해결될 때까지 반복

**결과**:
- 47개 사용처 모두 정확히 수정
- 타입 안전성 보장
- 토큰 소모 최소

*** 예시 2: 안전한 함수명 변경

**** 작업
#+begin_src python
# 기존
def process_user_data(user_data):
    ...

# 변경
def process_user_profile(user_data):
    ...
#+end_src

**** LSP 활용
1. ~get_references("process_user_data")~
   → 23개 호출 위치

2. 각 위치를 정확히 변경
   - 주석은 변경 안 함 (LSP는 코드만)
   - 문자열 리터럴도 변경 안 함

3. ~get_diagnostics~
   → 정의되지 않은 함수 오류 확인

**장점**:
- False positive 없음
- 안전한 리팩토링
- 수동 검토 불필요

- 토큰 제로, 컴파일러 정확도

2. OpenCode의 차별화 요소
3. Claude Code의 잠재적 약점
